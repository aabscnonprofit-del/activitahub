# ADR — Project View Architecture

> **Status: ACCEPTED (AUTHORITATIVE)** · **Date: 2026‑07‑05** · **Type: Architecture Decision Record.**
> This ADR establishes how every person interacts with a Project: **there is only one Project**, and different
> people receive different **role‑based Views (projections)** of it — never different Project models. It
> introduces no new entity, root, or Project model, and does not modify implementation; it is documentation
> only. Governed by `ALH_PRODUCT_PHILOSOPHY.md`; current architecture in `CURRENT_ARCHITECTURE.md`; builds on
> the roles decision in `ADR_010_HUMAN_ROLES_AND_PIPELINE_AUTHORITY.md` and the ownership rule in
> `ADR_003_ENTITY_OWNERSHIP.md`. Where this ADR and the axioms/specifications disagree, those documents win.

---

## Purpose

Define how every person interacts with a Project inside ActivLife Hub.

This decision establishes one of the core architectural principles of the platform:

**There is only one Project.**

Different people do not receive different project models.

Instead, they receive different **views (projections)** of the same Project.

---

## Core Principle

A Project exists only once.

The system never creates separate Organizer Projects, Participant Projects, Worker Projects, Client Projects, or Emergency Projects.

Instead, every interaction with a Project is performed through a **role-based Project View**.

---

## Fundamental Rule

**Project Views belong to the Project, not to the User.**

The system does not classify users permanently as organizers, participants, workers, or clients.

Instead, every person's role is evaluated independently for every Project.

The same user may simultaneously be:

* Organizer of Project A.
* Participant of Project B.
* Photographer of Project C.
* Client of Project D.

The Project determines which View is rendered.

---

## Standard Project Views

### Public View

Visible before registration.

Purpose:

* discover activities;
* browse the marketplace;
* decide whether to participate.

Contains only public information.

---

### Participant View

Visible to registered participants.

Contains information required for participation.

Examples include:

* ticket;
* QR code;
* event updates;
* schedule;
* arrival instructions;
* participant materials;
* certificates;
* participant communications.

---

### Organizer View

Complete operational view.

Contains all organizer capabilities.

---

### Client View

Visible to the customer who ordered the event.

The organizer explicitly controls which information is shared.

Clients may be attached through:

* invitation link;
* email;
* phone number;
* existing ActivLife Hub account.

Registration is optional until the Client chooses to claim an account.

---

### Worker View

Visible to assigned workers and service providers.

Examples:

* waiter;
* photographer;
* videographer;
* musician;
* DJ;
* entertainer;
* driver;
* coordinator;
* decorator;
* catering staff;
* equipment providers.

Workers receive only the information required to perform their assigned responsibilities.

---

### Emergency / Safety View

Accessible without registration.

Access is granted through secure scoped links generated by the organizer.

Typical users include:

* fire department;
* police;
* emergency medical services;
* venue security;
* safety inspectors;
* technical inspectors.

Emergency personnel must never be required to register before obtaining access to critical project information.

Only safety-related information is exposed.

Financial, commercial, organizer-only and unrelated participant information must never be exposed.

---

## Architectural Consequences

The platform contains:

* one Project;
* one Project data model;
* multiple Project Views.

Views never own business data.

Views are filtered projections of the same Project.

No duplicated Project entities may exist.

No duplicated business logic may exist between Views.

Permissions determine visibility.

---

## Relationship to Existing Architecture

```
Planning
    ↓
Project
    ↓
Role-based Project Views
    ├── Public View
    ├── Participant View
    ├── Organizer View
    ├── Client View
    ├── Worker View
    └── Emergency / Safety View
```

Organizer operational modules (Capacity Gate, Lead Organizer, Delivery, Team, Execution, Budget, Timeline, etc.) remain part of the Organizer View and are never duplicated into other Views.

---

## Design Principle

**One Project. Multiple Views. Zero duplicated Project models.**
