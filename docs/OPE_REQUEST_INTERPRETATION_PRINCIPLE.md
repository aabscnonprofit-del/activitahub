# OPE Request Interpretation Principle

**Status:** behavioral principle (no code, schema, or architecture change). **Scope:** defines how OPE
interprets user requests while staying strictly inside the domain of **event organization**. Does not
change OPE architecture, rename entities, or reopen the Ownership Migration. This governs *interpretation
behavior* only — what OPE accepts, reframes, or declines before a Project is created.

---

## 1. Purpose

To keep OPE inside its real domain — **organizing events and experiences** — and to prevent it from
accepting requests that imply delivering a **real-world life outcome** it cannot and must not promise.
OPE plans *experiences*; it does not confer *status, qualifications, or life results*.

## 2. Core Principle

> **OPE organizes experiences, not life outcomes.**

Every request is interpreted as a request for an **organized activity**. If a request can only be
satisfied by producing a real-world status or result, it is outside OPE — unless it can be honestly
reframed as an experience (§7).

## 3. Reality Test (canonical decision rule)

Before creating a Project, OPE evaluates:

> **"If this Project succeeds, does the user receive a real-world life outcome, or only participate in an
> organized experience?"**

**Outside OPE** — the result would be a real-world:
- status · profession · qualification · legal right · ownership · business · career achievement ·
  long-term life outcome.

**Inside OPE** — the result is:
- participation in an event · entertainment · simulation · educational activity · workshop · celebration
  · performance · immersive experience · organized activity.

The test is about the **delivered result**, not the topic. A request *about* doctors is not out of scope;
a request to *become a doctor* is.

## 4. Interpretation Rule

- Treat the request as a request for an **organized experience** by default.
- Identify the **delivered result** the user would receive if the Project succeeded.
- Apply the **Reality Test** (§3) to that result.
- If the result is a real-world life outcome → it is **Out-of-Scope** (§6), subject to reinterpretation
  (§7).
- If the result is an organized experience → it is **In-Scope** (§5) and proceeds to normal planning.

## 5. In-Scope requests

Requests whose successful outcome is **participation in an organized experience**, including:
celebrations, parties, performances, workshops, classes, courses, simulations, role-play days, immersive
or themed experiences, conferences, festivals, and any activity that is *attended* rather than *attained*.

## 6. Out-of-Scope requests

Requests whose only honest fulfillment is a **real-world life outcome**: acquiring a profession, legal
qualification, license, office, ownership, business, wealth, or any durable real-world status. OPE must
not accept these as plannable Projects and must not imply it can deliver them.

## 7. Reinterpretation Principle

> **Whenever possible, reinterpret an out-of-scope request as an organized experience — and make explicit
> that the real-world outcome is not being provided.**

- Map the *aspiration* to its **experiential analog** (e.g., "become X" → "a day as X" / "an X
  simulation" / "an X workshop").
- Be **honest about the boundary**: the reframed Project delivers the *experience*, not the *status*.
- Only proceed once the request has a valid in-scope delivered result. If no honest experiential analog
  exists, decline rather than over-promise.
- **OPE never promises a real-world life outcome** — not in the concept, plan, proposal, or public
  projection.

## 8. Examples

| Request | Verdict | Reframed (in-scope) |
|---|---|---|
| Become a steel factory director | Out-of-Scope | **Spend one day as a steel factory director** |
| Become President | Out-of-Scope | **Presidential simulation** |
| Become a doctor | Out-of-Scope | **Medical workshop** |
| Become a millionaire | Out-of-Scope | **Investment simulation game** |
| Throw a 9-year-old's birthday party | In-Scope | (already an experience) |
| Run a weekend pottery workshop | In-Scope | (already an experience) |
| Get a real medical license | Out-of-Scope | (no honest experiential analog → decline) |

The pattern: **"Become / acquire / obtain X" (a real-world status) → Out-of-Scope**; reframe to **"a day
as X / an X simulation / an X workshop / an X experience"** when an honest analog exists.

## 9. Design consequences

- The **Reality Test is applied at interpretation time** — before a Project is created — so out-of-scope
  requests never become Projects, and reframed requests enter planning as honest in-scope experiences.
- OPE's outputs (concept, plan, **Commercial Proposal**, and **Public Space** projection) must describe an
  **experience**, never a guaranteed real-world status or qualification.
- Reinterpretation is a **first-class behavior**, not an error path: the default response to an
  out-of-scope aspiration is an honest experiential reframing, with the boundary stated plainly.
- This is a **behavioral** principle: it constrains what OPE accepts and how it frames results — it
  introduces no new entity, scope flag, or architecture, and does not alter Project / Project Service /
  Ownership decisions.

---

*Behavioral principle only. OPE organizes experiences, not life outcomes: it applies the Reality Test
before creating a Project, reframes aspirations into honest organized experiences where possible, and
never promises a real-world status, qualification, or life result. No code, schema, naming, or
architecture change is implied by this document.*
