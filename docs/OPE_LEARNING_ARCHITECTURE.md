# OPE Learning Architecture — как OPE учится на реальных событиях

> **Тип:** архитектура обучения (без кода, без UI, без схемы БД).
> **Продолжает:** `OPE_PLANNING_WORKFLOW.md` (Part 3 корректировки, Part 4 обратная связь),
> `OPE_MASTER_SPEC.md` §13 (Monitoring Engine), `OPE_KNOWLEDGE_MODEL.md` (блоки знаний),
> `types.ts` (`HistoricalPriceRecord`), `ADR_002` (гейт честности).
> **Дата:** 2026-06-10.

## 0. Базовый принцип

OPE — детерминированный движок поверх **курируемого знания**. Обучение не «переучивает модель», а
**предлагает обновления знания**, которые проходят проверку перед применением. Зеркало frozen-field
guard: как движок не выдумывает факты в плане, так и обучение **не переписывает знание из одного
сигнала** — оно копит доказательства и продвигает их по уровням.

Три жёстких правила:
1. **Никакого «тихого» переписывания общего знания** одной корректировкой (кроме local-plan-only).
2. **Безопасность только ужесточается и только через эксперта** — никогда автоматически, никогда от
   одного организатора.
3. **Любое авто-обновление обратимо** и несёт провенанс (кто/сколько подтверждений/когда).

---

## 1. Какие сигналы собираются

| Сигнал | Что собираем | Откуда | Куда ведёт |
|---|---|---|---|
| **Corrections** (правки плана) | изменённые line-item цены, количества, добавленные/убранные задачи и риски, правки сообщений; **scope-тег** (plan / regional / pattern) | стадия Organizer review/corrections (workflow Part 3) | ценовые/правило-предложения по уровню scope |
| **Actual costs** (факт. расходы) | реальная стоимость по line-item после события | пост-событийный ввод организатора (опц. чеки) | **regional pricing** (`HistoricalPriceRecord`) |
| **Actual attendance** (факт. посещаемость) | приглашено / подтвердило / пришло; для recurring — посещаемость по сессиям | пост-событие + RSVP/участники | **no-show буферы, RSVP-ставки, retention** |
| **Actual incidents** (инциденты) | инциденты и near-miss по безопасности (травма, аллергия, погода, провал надзора) | пост-событийный отчёт организатора | **только в экспертную очередь** по safety-знанию |
| **Actual organizer changes** (что организатор реально менял) | какие задачи добавил/убрал/переставил относительно сгенерированного плана; на какие риски реагировал; сдвиги таймлайна | дельта «сгенерировано → проведено» | **pattern-skeleton / ratio предложения** |

Каждый сигнал фиксируется **с контекстом**: pattern, category, region (city/state/country), снимок
сценария (счётчики, venue, recurrence, instructor/materials), timestamp, organizer id, источник. Без
контекста сигнал бесполезен (нельзя агрегировать).

## 2. Три уровня знания и как формируются предложения

| Уровень | Что обновляет | Как формируется предложение | Продвижение |
|---|---|---|---|
| **Local** (этот план) | разовая истина для одного события («моя площадка бесплатна») | прямо из корректировки со scope=plan | применяется к плану, **никогда не обобщается** |
| **Regional** (город/регион) | ценовые bands, no-show ставки, локальные нормы для `(pattern, line-item, region)` | агрегация **actual costs / corrections / attendance** внутри региона | продвигается в **regional pricing/rule слой** после N независимых подтверждений + отсева выбросов |
| **Global / Pattern** (правило паттерна) | ratio/правило/скелет, **не зависящие от локации** («food ratio 1.25→1.30», «классам нужен шаг разогрева») | кросс-региональная корроборация organizer-changes/corrections | **только экспертное ревью**; высший порог |

Поток формирования: **сигнал → запись с контекстом → агрегация по `pattern×region×line-item/rule` →
кандидат-предложение с confidence → маршрут по классу обновления (§3) → продвижение или очередь**.

## 3. Что обновляется как — авто / организатор / эксперт

Это ядро ответа. Класс обновления зависит от **риска ошибки**, **обратимости** и **затронутой
аудитории**.

### 3.1 Обновляется АВТОМАТИЧЕСКИ (статистика, обратимо, низкий вред)
> авто = по достижении порога подтверждений + отсев выбросов; всегда обратимо, всегда с провенансом.

- **Regional pricing bands** из actual costs — деньги статистичны и обратимы; авто-обновление
  регионального слоя при N корроборациях. (`HistoricalPriceRecord` → historical pricing provider.)
- **No-show / RSVP ставки** из attendance — авто-тюнинг региональных/паттерновых буферов.
- **Не-safety количественные коэффициенты в пределах узкого клампа** (напр. напитки на гостя ±10%) —
  авто только внутри жёсткой границы и при корроборации; больший сдвиг → организатор/эксперт.
- **Recurring retention-статистика** (отток по сессиям) — авто, как справочный сигнал.

### 3.2 Требует ПОДТВЕРЖДЕНИЯ ОРГАНИЗАТОРА
> организатор, проводивший событие, подтверждает, что факт реален (не опечатка), и выбирает scope.

- **Каждая корректировка при сабмите**: «применить к этому плану / предложить для моего региона /
  предложить как общее» — организатор задаёт scope. Региональное предложение засчитывается только после
  подтверждения, что actual реален.
- **Добавление/удаление задач и не-safety структуры плана** — предложение из organizer-changes;
  продвигается лишь после подтверждений от **нескольких** организаторов в регионе/паттерне.
- **Локальные venue/vendor специфики** (название площадки, контакт) — подтверждаются организатором,
  остаются на regional уровне.

### 3.3 Требует ЭКСПЕРТНОГО ПОДТВЕРЖДЕНИЯ
> доменный эксперт / ревьюер безопасности; никогда не авто, никогда от одного организатора.

- **Любое safety/risk-знание**: инциденты, ratio надзора за детьми, severity рисков, митигации,
  never-drop пункты. Безопасность **только ужесточается** и только через эксперта.
- **Legal / permits / regulated** знание.
- **Global / pattern правила** (кросс-региональные ratio, изменения скелета паттерна) — влияют на все
  планы → высший порог.
- **Всё, что касается несовершеннолетних/уязвимых** (safeguarding [S]).
- **Логика ценообразования между регионами** (не bands, а правила резолвинга/валюты).

### Сводная матрица

| Тип данных | Авто | Организатор | Эксперт |
|---|:--:|:--:|:--:|
| Региональные ценовые bands (actual cost) | ● | (подтверждение факта) | — |
| No-show / RSVP буферы | ● | — | — |
| Не-safety ratio (узкий кламп) | ● | при большем сдвиге | — |
| Добавление/удаление задач плана | — | ● | при глобализации |
| Venue/vendor специфика | — | ● | — |
| Pattern-level ratio (глобально) | — | — | ● |
| Risk/safety знание, инциденты | — | — | ● |
| Supervision/minors/safeguarding | — | — | ● |
| Legal / permits / regulated | — | — | ● |

## 4. Пороги продвижения (promotion thresholds)

- **Local:** мгновенно (только этот план).
- **Regional:** требуется **N независимых** подтверждающих точек (corrections **или** actuals) в этом
  регионе для `(pattern, line-item)`, с **отсевом выбросов** и **взвешиванием по свежести** (decay).
  Один организатор не двигает регион.
- **Global / Pattern:** кросс-региональная корроборация **+ экспертная подпись**. Никогда по данным
  одним.
- **Safety:** порог не статистический — **экспертное решение**; единичный серьёзный инцидент может
  ужесточить знание сразу (через эксперта), но не ослабить его.

Каждое предложение несёт **confidence** = функция (число источников, их независимость, свежесть,
согласованность, отсутствие выбросов).

## 5. Защита целостности (anti-poisoning)

- **Evidence-promoted, не single-source** — кроме local-plan-only.
- **Отсев выбросов** — резко отклоняющийся actual не двигает band.
- **Независимость источников** — N подтверждений должны идти от **разных** организаторов/событий;
  повторы от одного не суммируются.
- **Свежесть + decay** — старые данные весят меньше; знание «дышит» вместе с рынком.
- **Provenance / audit trail** — у каждого обновления: кто, сколько корробораций, когда, какой источник
  (`source: seed | actual | correction:<org> | provider`).
- **Асимметрия безопасности** — safety-знание может авто-**предлагаться** только в сторону строгости;
  применяется только экспертом; никогда не ослабляется по данным.
- **Обратимость / rollback** — любое авто-обновление откатываемо; если последующие данные расходятся,
  слой откатывается и помечается.
- **Frozen-field-guard аналог** — обучение обновляет **KB-слой**, но **никогда не переписывает уже
  зафиксированные факты конкретного плана**.

## 6. Как выученное возвращается в движок

- **Regional pricing** → `HistoricalPricingProvider` в резолвер-цепочке (`local → historical → external →
  fallback`): следующий план в этом городе менее «fallback», точнее по валюте/ценам.
- **Ratio / буферы** → конфиги derived-quantities и no-show буферов (Resource/Schedule блоки).
- **Retention** → знание Recurring/Community (когда модификатор появится).
- **Safety** → KB рисков **только после эксперта**; всегда в сторону большей осторожности.
- **Pattern-skeleton** → задачи/таймлайн паттерна **только после эксперта** (global).

Это замыкает петлю Monitoring (`OPE_MASTER_SPEC §13`): actuals → store → лучшее Budget/Resource на
следующем запросе; safety и pattern меняются **только ревью**, не данными.

## 7. Конвейер обучения (стадии)

```
1 Capture     сигнал + контекст (pattern, region, scenario snapshot, org id, ts)
2 Validate    границы здравого смысла, дедуп, отсев явных ошибок
3 Aggregate   группировка по pattern × region × line-item/rule
4 Propose     при достижении порога → кандидат-обновление + confidence
5 Route       класс: авто / организатор / эксперт (§3)
6 Promote     применить к нужному слою  ИЛИ  в очередь подтверждения
7 Re-enter    provider/конфиг/retention → влияет на следующий план
8 Monitor     расхождение → rollback + флаг
```

---

## 8. Что НЕ обучается данными — никогда

- Безопасность/надзор/minors — только эксперт, только строже.
- Legal/regulated — только эксперт.
- Coverage-гейт и отказы (ADR_002) — меняются решением, не статистикой (иначе данные могли бы «открыть»
  опасный паттерн).
- Конкретные факты плана (дата, локация, имена) — это вход, не знание.

---

## Итог

OPE учится **тремя контурами**: авто (региональные цены/буферы — статистично, обратимо), организатор
(структура плана, региональные специфики — подтверждение реальности и scope), эксперт (безопасность,
legal, глобальные правила паттернов — высший порог). Данные **поднимаются по уровням** (local → regional
→ global) только по доказательствам, безопасность движется только к строгости и только через эксперта, а
конкретные планы и гейт честности данными не переписываются. Так знание становится точнее со временем,
не теряя доверия.
