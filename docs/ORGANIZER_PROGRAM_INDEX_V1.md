## Section 1 — Program Overview

The **Organizer Program** is the complete set of documents that define how
ActivLife Hub trains, tests, and certifies organizers. It exists so that a person
can reliably take a client's vague **dream** and turn it into a delivered event —
the core chain the whole program teaches:

```
Dream → Concept → OPE Plan → Event → Follow-up
```

There are **two organizer statuses** and **two routes** to the higher one:

- **Verified Organizer** — a trust-and-identity status (a real, accountable
  person). Not a measure of craft.
- **Certified Organizer** — proof of the full organizer craft. Reachable **two
  ways**, which both grant the same status:
  1. Completing the **training path** (Section 3), or
  2. Passing the **experienced-organizer competency exam** (Section 4).

The program is organized into four layers, each its own document set:

1. **Foundation** — the principles, roles, and definitions everything rests on.
2. **Competencies** — the nine things an organizer must be able to do.
3. **Training path** — the structure and the twelve modules that teach the
   competencies to a beginner.
4. **Experienced-organizer certification path** — the exam structure and case bank
   that let an already-experienced organizer prove the competencies without
   training.

This index covers Sections 2–4 (the documents), Section 5 (how they connect),
Section 6 (which document is canonical for each area), and Section 7 (status).

## Section 4 — Experienced Organizer Certification Path (route 2 to Certified)

For a person who **already organizes real events** and wants to prove the craft
without the training. Tests the same competencies the training builds toward.

| Document | What it is |
|---|---|
| **VERIFIED_ORGANIZER_EXAM_STRUCTURE_V1** | The exam structure: nine sections (A–I), what each tests, evaluation logic (passing/failure philosophy, disqualifiers, auto vs. human review). **Source of truth for the exam.** |
| **CERTIFIED_ORGANIZER_EXAM_CASES_V2** | The case bank: 40 practical scenarios across the nine sections, tagged Judgment vs. Gate. **Source of truth for the case bank.** |
| **CERTIFIED_ORGANIZER_CASE_AUDIT_V1** | The audit of the V1 cases that drove the revisions into V2. **Supporting** (rationale/history), not the live bank. |

**Naming note (important for reviewers):** the exam-structure file is named
`VERIFIED_ORGANIZER_EXAM_STRUCTURE_V1` for historical reasons, but the status it
grants is **Certified Organizer**, not Verified. The "Verified" in the filename is
historical; the document itself states this. *(If you were looking for a file named
`CERTIFIED_ORGANIZER_EXAM_STRUCTURE_V1`, this is it.)*

**Superseded (historical context only):**
- `CERTIFIED_ORGANIZER_EXAM_CASES_V1` — the first 31-case draft, **superseded by** `..._V2` (40 cases). The audit (`..._CASE_AUDIT_V1`) is what bridged V1 → V2.

## Section 5 — Document Dependency Map

How the documents build on each other (arrow = "is built on / derived from"):

```
            ┌─────────────────────────────────────────────┐
            │  ORGANIZER_LEARNING_STRUCTURE_V1            │  (superseded →
            │  ORGANIZER_COMPETENCY_REQUEST_TO_OPE        │   folded in)
            └───────────────────┬─────────────────────────┘
                                │  consolidated into
                                ▼
        ┌───────────────────────────────────────────────┐
        │  ACTIVLIFE_HUB_ORGANIZER_FOUNDATION_V1         │◄────────┐
        │  ACTIVLIFE_HUB_ORGANIZER_COMPETENCIES_V1       │         │
        └───────┬───────────────────────────┬───────────┘         │
                │                            │                     │
   builds on    │                            │  builds on          │  all reference
                ▼                            ▼                     │
 ┌───────────────────────────┐   ┌──────────────────────────────┐ │
 │ CERTIFIED_ORGANIZER_       │   │ VERIFIED_ORGANIZER_          │ │
 │ TRAINING_STRUCTURE_V1      │   │ EXAM_STRUCTURE_V1            │ │
 └────────────┬──────────────┘   └──────────────┬───────────────┘ │
              │ defines modules                 │ defines sections │
              ▼                                 ▼                  │
 ┌───────────────────────────┐   ┌──────────────────────────────┐ │
 │ MODULE_1_FINAL_V1          │   │ CERTIFIED_ORGANIZER_         │ │
 │ MODULE_2 … MODULE_12       │   │ EXAM_CASES_V2 ◄── CASE_      │ │
 │ (_CONTENT_V1)             │   │            AUDIT_V1 (V1→V2)   │ │
 └────────────┬──────────────┘   └──────────────┬───────────────┘ │
              │                                  │                 │
              └──────────────── both ────────────┴─────────────────┘
                         reference the case bank + foundation
```

Key relationships in words:

- **Foundation + Competencies** are the root. Both the training path and the exam
  path derive from them.
- **Training Structure** defines the 12 modules; each **module's content** is built
  from the structure, the Foundation, the Competencies, and the **case bank** (the
  modules teach toward the same situations the exam tests).
- **Exam Structure** defines the nine sections; the **case bank (V2)** populates
  them. The **case audit** is the bridge that turned the V1 draft into V2.
- The two paths **converge**: both lead to the same **Certified Organizer** status.

## Section 7 — Program Status

**Complete:**

- ✅ **Foundation** — consolidated and reconciled (`FOUNDATION_V1`).
- ✅ **Competencies** — nine defined, mapped to Verified/Certified (`COMPETENCIES_V1`).
- ✅ **Training structure** — 12-module path defined (`TRAINING_STRUCTURE_V1`).
- ✅ **Training content** — all 12 modules written, covering every competency
  (C1–C9) plus the integrated capstone. Module 1 has been through a full
  draft → review → final cycle.
- ✅ **Exam structure** — nine sections + evaluation logic (`EXAM_STRUCTURE_V1`).
- ✅ **Case bank** — 40 cases, audited and revised to V2 (`EXAM_CASES_V2`).
- ✅ **Dual-path certification** — both routes to Certified defined and reconciled
  across the Foundation and exam docs.

**Known gaps / not yet done (factual — no new work proposed here):**

- ⏳ **Module review cycle** — only Module 1 has been formally reviewed and finalized
  (`MODULE_1_FINAL_V1`). Modules 2–12 exist as single `_CONTENT_V1` drafts and have
  not yet been through an equivalent beginner-review → final pass.

**Deliberately out of scope (by instruction, not oversight):**

- 🚫 **Certification rules** (how a pass is awarded, thresholds, validity) — not
  defined in any program document by design.
- 🚫 **Scoring** — the exam and cases define *what* is evaluated and the
  pass/fail/disqualify logic, but not numeric scores.
- 🚫 **Exam questions** — the case bank holds scenarios, not finished question items.
- 🚫 **Pricing, platform UI, and delivery mechanics** — the program defines content
  and judgment, not how it's priced or surfaced in the product.

## How to use this index

- **New developer/reviewer:** read Section 1, then the two Foundation docs (Section
  2). That's enough to understand the whole program's intent.
- **Working on training:** start at `TRAINING_STRUCTURE_V1`, then the relevant
  module (remember Module 1's canonical file is the **Final**).
- **Working on certification/exam:** start at `VERIFIED_ORGANIZER_EXAM_STRUCTURE_V1`,
  then `CERTIFIED_ORGANIZER_EXAM_CASES_V2`.
- **When in doubt about which file is authoritative:** use the Source-of-Truth Map
  (Section 6).

*This index is organizational only and introduces nothing new. If it ever conflicts
with a canonical document named in Section 6, the canonical document is correct and
this index should be updated.*
